## Этап 1 — DeepSeek интеграция

### 1.1 Системные промпты

**Чат-ассистент:**
- Роль: финансовый ассистент пользователя
- Контекст: язык пользователя, валюта, краткая сводка баланса/трат (передаётся из бэкенда)
- Ограничения: не даёт медицинских/юридических советов, не запрашивает пароли/пин-коды
- Формат ответа: структурированный (markdown), локализованный

**Голосовой ввод (voice-to-expense):**
- Роль: парсер устной речи в структурированную транзакцию
- Вход: распознанный текст (STT на клиенте → текст на бэкенд)
- Выход: JSON `{ amount, currency, category, description, type }`
- Fallback: если не удалось распознать — вернуть текстовое уточнение

**Распознавание чека (receipt-to-expense):**
- Роль: парсер фотографии чека в список транзакций
- Вход: текст, распознанный OCR (на клиенте или сервере)
- Выход: JSON `[{ amount, description, category }]`
- Fallback: если чек не распознан — попросить ручной ввод

### 1.2 Защита от перезаписи (prompt injection)

- Системный промпт передаётся как `system` роль, пользовательский — `user`
- Валидация на бэкенде: длина сообщения ≤ 2000 символов
- Фильтрация: запрет ключевых фраз (`ignore previous`, `you are now`, `system:`)
- Rate limiting: на уровне API-ендпоинтов, не на уровне модели

### 1.3 Rate limits

| План | Чат (сообщений/день) | Голос (запросов/день) | Чеки (сканов/день) |
|------|----------------------|-----------------------|---------------------|
| Free | 5 | 3 | 2 |
| Premium | 50 | 20 | 10 |

- Implement: Redis-счётчики с TTL до конца суток
- Ключ: `rl:{userId}:{feature}:{date}`
- При превышении — HTTP 429 с `Retry-After`
- Счётчики сбрасываются в 00:00 UTC

### 1.4 Backend

- Новый модуль `AiModule` с сервисом `AiService`
- Эндпоинты: `POST /api/ai/chat`, `POST /api/ai/voice`, `POST /api/ai/receipt`
- DeepSeek API: модель `deepseek-chat`, температура 0.3 для структурированных ответов, 0.7 для чата
- Контекст пользователя (баланс, категории, последние траты) инжектится в системный промпт
- Streaming для чата (SSE)

## Этап 2 — Фремиум-модель

### 2.1 Разделение фич

**Бесплатно (Free):**
| Фича | Лимит |
|-------|-------|
| Транзакции | без лимита (добавление, просмотр, редактирование) |
| Категории | без лимита |
| Счёта | до 3 счетов |
| Базовая аналитика | месячная сводка, график трат по категориям |
| Статьи | чтение до 3 статей/день |
| AI-чат | 5 сообщений/день |
| Голосовой ввод | 3 запроса/день |
| Сканирование чеков | 2 скана/день |

**Премиум (Premium):**
| Фича | Описание |
|-------|----------|
| Безлимитный AI-чат | 50 сообщений/день |
| Голосовой ввод | 20 запросов/день |
| Сканирование чеков | 10 сканов/день |
| Расширенная аналитика | тренды, прогнозы, сравнение периодов |
| Цели (Goals) | постановка и отслеживание финансовых целей |
| Вишлист | безлимитный список желаний с автоотложением |
| Стоимость жизни (Life Cost) | калькулятор стоимости жизни по городам |
| Безлимит аккаунтов | более 3 счетов |
| Безлимит статей | чтение без дневного лимита |
| Экспорт данных | CSV/PDF выгрузка |
| Ранний доступ | новые фичи первыми |

### 2.2 Модель подписки

| Период | Цена |
|--------|------|
| 1 месяц | $2.99 / 299₽ |
| 12 месяцев | $19.99 / 1999₽ (44% скидка) |

- Подписка привязана к аккаунту (не к устройству)
- Trial: 7 дней бесплатного Premium при первой подписке
- Отмена: доступ до конца оплаченного периода

### 2.3 Backend

- Prisma: модель `Subscription { userId, plan: FREE|PREMIUM, expiresAt, stripeSessionId? }`
- Middleware `PremiumGuard` — проверяет `req.user.plan === 'PREMIUM'` и срок действия
- При просрочке — автоматически падает в FREE
- Эндпоинт `GET /api/subscription` — текущий план, даты, лимиты
- Эндпоинт `GET /api/subscription/limits` — оставшиеся лимиты по фичам

## Этап 3 — Переработка авторизации

### 3.1 Способы авторизации

**Telegram Login:**
- Widget на клиенте → проверка `hash` на бэкенде через `SHA256(bot_token, auth_data)`
- Создание пользователя: `telegramId`, `username`, `avatar`
- Привязка: если юзер уже зарегистрирован (email) — связать аккаунты

**Magic Link (email):**
- `POST /api/auth/magic` → отправка письма с JWT-ссылкой
- `GET /api/auth/magic/verify?token=...` → авторизация, выдача access+refresh
- Ссылка действительна 15 минут
- Rate limit: 5 запросов/час на email

**Пароль (текущий):**
- Оставить как fallback
- Отдельный флоу: email + пароль
- Валидация: минимум 8 символов, не в списке утечек (haveibeenpwned API)

### 3.2 Сессии и токены

- Access token: JWT, 15 минут
- Refresh token: JWT, 30 дней, rotatable (новый при каждом использовании)
- Хранение: Access в памяти, Refresh в SecureStore (mobile) / httpOnly cookie (web)
- Эндпоинты: `POST /api/auth/refresh`, `POST /api/auth/logout`

### 3.3 Флоу на мобильном

- Экран 1: Выбор способа (Telegram / Email magic link / Пароль)
- Telegram: открывает WebView/Bot → callback → авторизация
- Email: ввод почты → письмо с magic link → deep link в приложение
- Пароль: ввод почты + пароль → JWT

### 3.4 Backend

- Модуль `AuthModule`
- Стратегии: `LocalStrategy`, `TelegramStrategy`, `MagicLinkStrategy`
- `POST /api/auth/telegram` → проверка hash, upsert user
- `POST /api/auth/magic` → отправка письма
- `GET /api/auth/magic/verify` → верификация токена
- `POST /api/auth/login` → email + пароль
- `POST /api/auth/register` → email + пароль
- `POST /api/auth/refresh` → refresh токен

## Этап 4 — Платежи

### 4.1 Структура

```
Mobile App
  ├── iOS → StoreKit 2 (in-app purchase)
  ├── Android → Google Play Billing Library
  ├── Huawei → AppGallery IAP
  └── RuStore → RuStore Billing SDK

Backend
  └── /api/subscription
       ├── POST /validate    ← прием server-to-server уведомлений
       ├── POST /restore    ← восстановление покупок
       └── GET  /status     ← текущий статус подписки
```

### 4.2 iOS — App Store

- StoreKit 2 (async/await API)
- Продукт: `com.moneytracker.premium.monthly`, `com.moneytracker.premium.yearly`
- `Transaction.updates` — слушатель обновлений
- Валидация: бэкенд проверяет `appReceipt` через Apple Server API
- Sandbox-тестирование на этапе разработки

### 4.3 Android — Google Play

- Google Play Billing Library 7+
- Продукт: `premium_monthly`, `premium_yearly`
- `PurchasesUpdatedListener` — обработка покупок
- Валидация: бэкенд проверяет `purchaseToken` через Google Play Developer API
- `acknowledgePurchase` после серверной валидации

### 4.4 Huawei — AppGallery

- Huawei IAP Kit
- Продукт: `premium_monthly`, `premium_yearly`
- Валидация: бэкенд проверяет `InAppPurchaseData` + подпись через Huawei REST API
- Поддержка: только устройства HMS (Huawei Mobile Services)

### 4.5 RuStore

- RuStore Billing SDK
- Продукт: `premium_monthly`, `premium_yearly`
- Валидация: бэкенд проверяет через RuStore API
- `PurchaseStateAnalyzer` — обработка состояний покупки

### 4.6 Backend

- Модуль `PaymentsModule`
- `POST /api/payments/validate` — валидация чека/токена от стора
- `POST /api/payments/restore` — восстановление покупок
- `GET /api/payments/products` — список доступных продуктов
- Webhook endpoints для server-to-server уведомлений от каждого стора
- Логирование: `Payment { userId, platform, productId, transactionId, status, createdAt }`

- Фильтры: по категории, счёту, типу (доход/расход)

## Этап 7 — Тестирование и правки

### 7.1 Типы тестирования

- Unit-тесты: сервисы бэкенда (Jest), утилиты мобильного приложения
- Integration-тесты: API-эндпоинты (supertest)
- E2E-тесты: критические сценарии (Maestro/Detox)
- Ручное тестирование: все флоу на iOS/Android

### 7.2 Критические сценарии для E2E

1. Регистрация → логин → создание транзакции → просмотр
2. Подписка → проверка премиум-фич → отмена → проверка падения в free
3. AI-чат → hit rate limit → показ paywall
4. Сканирование чека → создание транзакций
5. Голосовой ввод → создание транзакции
6. Смена языка → проверка всех экранов

### 7.3 Чеклист перед релизом

- [ ] Все переводы заполнены (5 языков)
- [ ] App Store Review Guidelines compliance
- [ ] Google Play Policy compliance
- [ ] Privacy Policy опубликована
- [ ] App signing ключи в защищённом хранилище
- [ ] Sentry/ Crashlytics подключены
- [ ] Аналитика событий (Amplitude/Firebase)
- [ ] Rate limiter работает на всех AI-эндпоинтах
- [ ] Подписки восстанавливаются при переустановке
